# 时间戳更新机制说明

## 当前状态

### ✅ 已使用时间戳的地方

1. **文档元数据记录**
   - `created_at`: 文档创建时间
   - `updated_at`: 文档更新时间
   - 用于记录和审计，**不用于冲突解决**

2. **事件时间戳**
   - WebSocket消息中的 `timestamp` 字段
   - 用于记录事件发生时间

### ❌ 未使用时间戳的地方

**冲突解决机制**：
- 当前主要使用**版本号（version）**进行冲突检测
- 使用**差异合并（diff-based merging）**进行冲突解决
- **没有使用时间戳进行"最后写入获胜"（last-write-wins）策略**

## 当前冲突解决机制

### 1. 版本号检测
```python
if client_version < server_version:
    # 检测到冲突，需要合并
```

### 2. 差异合并
```python
# 使用 base_content 计算差异
merged = await self._merge_with_diff(
    base_content=base_content,
    client_content=client_content,
    server_content=server_content
)
```

### 3. 智能合并策略
- HTML内容：块级合并，保留所有唯一块
- 文本内容：行级合并，保留所有唯一行
- 删除操作：优先保留删除操作

## 是否需要时间戳更新机制？

### 优点
1. **简单直接**：最后写入的版本直接覆盖
2. **性能好**：不需要复杂的合并逻辑
3. **适合简单场景**：对于非协作编辑场景更合适

### 缺点
1. **数据丢失风险**：可能覆盖其他用户的更改
2. **不适合协作编辑**：多人同时编辑时容易丢失数据
3. **用户体验差**：用户可能发现自己的更改被覆盖

## 建议

### 当前方案（推荐）✅
- **版本号 + 差异合并**：适合协作编辑，不会丢失数据
- **智能合并**：保留所有用户的更改

### 可选方案
如果需要添加时间戳机制，可以：

1. **混合策略**：
   - 优先使用差异合并
   - 如果合并失败，使用时间戳（最后写入获胜）作为备用

2. **时间戳辅助**：
   - 使用时间戳判断更新顺序
   - 但仍然进行合并，而不是直接覆盖

3. **用户选择**：
   - 让用户选择冲突解决策略
   - 差异合并 vs 最后写入获胜

## 总结

**当前系统：**
- ✅ 使用版本号进行冲突检测
- ✅ 使用差异合并进行冲突解决
- ✅ 使用时间戳记录元数据（不用于冲突解决）

**如果需要时间戳机制：**
- 可以添加作为备用策略
- 或者作为用户可选的冲突解决方式
- 但不建议完全替换当前的差异合并机制
