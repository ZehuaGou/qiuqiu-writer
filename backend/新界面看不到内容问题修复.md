# 新界面看不到内容问题修复

## 问题描述

用户A编辑内容后，用户B打开新界面看不到A写的内容。

## 根本原因

1. **`getDocument()` 优先使用本地缓存**：新界面从缓存加载，但缓存可能没有A的内容
2. **A的内容可能未同步到服务器**：A编辑后只保存到本地缓存，没有及时同步到服务器
3. **服务器获取失败时使用缓存**：如果服务器获取失败，直接使用本地缓存（可能是空的）

## 修复方案

### 1. 修改 `getDocument()` - 优先从服务器获取

**修复前**：
- 先从本地缓存获取
- 如果在线，异步检查更新
- 缓存可能没有A的内容

**修复后**：
- **总是先尝试从服务器获取最新版本**
- 只有服务器获取失败或没有数据时，才使用本地缓存
- 使用缓存时，标记版本为更小的值，强制在同步时检查服务器

```typescript
// 关键修复：总是先尝试从服务器获取最新版本
let serverDoc = await this.fetchFromServer(documentId);
if (serverDoc) {
  // 使用服务器最新数据
  return serverDoc;
}

// 只有服务器获取失败时，才使用本地缓存
// 但标记版本为更小的值，强制在同步时检查服务器
```

### 2. 确保内容及时同步

- `updateDocument()` 已使用防抖同步
- 但需要确保在关闭前同步完成
- 可以考虑添加页面关闭前的强制同步

## 修复效果

### 修复前
1. 用户A编辑内容 → 保存到本地缓存
2. 用户A关闭浏览器 → 内容可能未同步到服务器
3. 用户B打开新界面 → 从服务器获取，但服务器没有A的内容
4. **结果**：B看不到A的内容 ❌

### 修复后
1. 用户A编辑内容 → 保存到本地缓存 + 防抖同步到服务器
2. 用户A关闭浏览器 → 内容已同步到服务器（防抖延迟内）
3. 用户B打开新界面 → **从服务器获取最新数据**
4. **结果**：B能看到A的内容 ✅

## 关键改进点

### 1. 优先服务器策略
- `getDocument()` 总是先尝试从服务器获取
- 只有服务器不可用时，才使用本地缓存
- 使用缓存时，标记为"可能过期"

### 2. 及时同步
- `updateDocument()` 使用防抖同步
- 确保内容在合理时间内同步到服务器
- 可以考虑添加页面关闭前的强制同步

### 3. 版本检查
- 使用缓存时，版本号减1，强制在同步时检查服务器
- 确保不会用旧数据覆盖新数据

## 测试场景

### 场景1：用户A编辑后，用户B打开
1. 用户A编辑第一段，保存
2. 等待1-2秒（防抖延迟）
3. 用户B打开新界面
4. **预期**：B能看到A的内容

### 场景2：用户A快速关闭
1. 用户A编辑内容，立即关闭浏览器
2. 用户B打开新界面
3. **预期**：如果A的内容已同步，B能看到；如果未同步，B看不到（这是正常的，因为A关闭太快）

### 场景3：离线编辑
1. 用户A离线编辑内容
2. 用户A恢复网络，同步
3. 用户B打开新界面
4. **预期**：B能看到A的内容

## 进一步优化建议

### 1. 页面关闭前强制同步

```typescript
// 在页面关闭前强制同步
window.addEventListener('beforeunload', async () => {
  // 强制同步所有待同步的文档
  await forceSyncAll();
});
```

### 2. 定期同步

```typescript
// 定期同步，确保内容及时保存
setInterval(() => {
  syncPendingDocuments();
}, 5000); // 每5秒同步一次
```

### 3. 同步状态提示

```typescript
// 显示同步状态，让用户知道内容是否已保存
if (syncInProgress) {
  showSyncIndicator('正在同步...');
} else if (hasPendingChanges) {
  showSyncIndicator('待同步');
} else {
  showSyncIndicator('已同步');
}
```

## 注意事项

1. **网络延迟**：服务器获取有2秒超时，避免阻塞太久
2. **离线支持**：服务器不可用时，仍可使用本地缓存
3. **同步延迟**：防抖同步有1秒延迟，确保内容及时保存
4. **用户体验**：如果A关闭太快，B可能看不到A的内容（这是正常的）
