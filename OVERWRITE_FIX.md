# 覆盖问题修复说明

## 问题描述

如果用户B没有及时获取更新，会将用户A的变更覆盖掉。

## 根本原因

1. **前端获取失败时使用旧版本号**：如果用户B在同步前获取服务器文档失败或超时，可能使用旧的版本号，导致后端认为没有冲突
2. **后端版本检查不够严格**：如果客户端版本号 >= 服务器版本号，后端可能直接更新，不进行合并检查
3. **并发修改检测不足**：即使版本号相同，如果内容不同（并发修改），也应该进行合并

## 已实施的修复

### 1. 后端：强制合并检查 ✅

**位置**: `backend/src/memos/api/services/sharedb_service.py`

**改进**:
- 即使版本号相同但内容不同，也会进行合并
- 检测并发修改（版本相同但内容不同）
- 确保所有内容差异都会触发合并逻辑

**关键代码**:
```python
# 关键改进：即使版本号相同或客户端更大，也要检查内容是否不同
content_different = server_content != content

# 如果服务器版本更新，或者内容不同，都需要合并
if server_version > version or (content_different and server_version == version):
    # 进行合并
    merged_content = await self._smart_merge_content(...)
elif content_different:
    # 版本号相同但内容不同（可能是并发修改），也需要合并
    merged_content = await self._smart_merge_content(...)
```

### 2. 前端：保守版本号策略 ✅

**位置**: `frontend/src/utils/sharedbClient.ts`

**改进**:
- 设置获取超时（2秒），避免阻塞
- 如果获取失败，使用更保守的版本号（减1），强制后端进行合并检查
- 检测并发修改（版本相同但内容不同）

**关键代码**:
```typescript
// 设置超时，避免阻塞太久
const fetchPromise = this.getDocument(documentId);
const timeoutPromise = new Promise<null>((resolve) => 
  setTimeout(() => resolve(null), 2000) // 2秒超时
);

serverDoc = await Promise.race([fetchPromise, timeoutPromise]);

// 如果获取失败，使用更保守的版本号，强制后端合并
if (fetchFailed) {
  syncVersion = Math.max(0, currentVersion - 1);
  console.log('⚠️ [同步] 使用保守版本号，强制后端合并检查:', syncVersion);
}
```

## 工作流程

### 场景1：用户B获取更新失败

1. **用户A**：同步成功，版本变成 2
2. **用户B**：尝试获取服务器文档，但超时或失败
3. **用户B**：使用保守版本号（当前版本 - 1 = 0）
4. **后端**：检测到 `server_version (2) > client_version (0)`，进行合并
5. **结果**：✅ 用户A的更改被保留，用户B的更改也被合并

### 场景2：用户B使用旧版本号

1. **用户A**：同步成功，版本变成 2
2. **用户B**：本地版本号是 1，但获取失败，使用保守版本号 0
3. **后端**：检测到 `server_version (2) > client_version (0)`，进行合并
4. **结果**：✅ 两个用户的更改都被保留

### 场景3：并发修改（版本相同但内容不同）

1. **用户A和B**：都从版本 1 开始编辑
2. **用户A**：先同步，版本变成 2
3. **用户B**：获取到版本 2，但本地内容已修改
4. **用户B**：同步时，版本号是 2（和服务器相同），但内容不同
5. **后端**：检测到 `server_version (2) == client_version (2)` 但 `content_different = true`，进行合并
6. **结果**：✅ 两个用户的更改都被保留

## 关键改进点

1. **后端总是检查内容差异**：即使版本号相同，如果内容不同也会合并
2. **前端保守策略**：获取失败时使用更低的版本号，确保后端检测到冲突
3. **超时机制**：避免获取操作阻塞太久
4. **并发修改检测**：识别版本相同但内容不同的情况

## 如果仍然覆盖

### 检查清单

1. ✅ 后端日志是否显示"检测到版本冲突"或"检测到并发修改"？
2. ✅ 前端是否显示"使用保守版本号"？
3. ✅ 合并后的内容长度是否合理？
4. ✅ 是否所有唯一内容都被保留？

### 进一步调试

如果问题仍然存在，可能需要：

1. **检查合并逻辑**：查看 `_smart_merge_content` 是否正确执行
2. **检查内容格式**：确认 HTML 格式是否被正确解析
3. **检查版本号更新**：确认版本号是否正确递增
4. **考虑操作级同步**：对于复杂场景，可能需要实现操作级同步（OT）
