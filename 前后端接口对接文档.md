# 前后端接口对接文档

## 📋 概述

本文档说明前后端接口的对接情况和使用方法。

## 🔗 接口基础配置

### 后端服务
- **默认地址**: `http://localhost:8001`
- **API文档**: `http://localhost:8001/docs`
- **OpenAPI规范**: `http://localhost:8001/openapi.json`

### 前端配置
- **环境变量**: `VITE_API_URL` (默认: `http://localhost:8001`)
- **配置文件**: `frontend/.env` 或 `frontend/.env.local`

```bash
# frontend/.env
VITE_API_URL=http://localhost:8001
```

## ✅ 已对接的接口

### 1. AI分析接口

#### 1.1 章节分析
- **前端调用**: `POST /ai/analyze-chapter`
- **后端实现**: ✅ 已实现
- **功能**: 对小说章节内容进行AI分析，返回结构化的章节分析结果

**请求示例**:
```typescript
const response = await fetch(`${API_BASE_URL}/ai/analyze-chapter`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    content: "章节内容...",
    prompt: "分析提示词...",
    settings: {
      model: "codedrive-chat",
      temperature: 0.7,
      max_tokens: 4000
    }
  }),
});
```

**响应格式** (流式响应):
```
data: {"type": "text", "text": "分析内容..."}
data: {"type": "metadata", "metadata": {...}}
data: {"type": "done"}
```

#### 1.2 健康检查
- **前端调用**: `GET /ai/health`
- **后端实现**: ✅ 已实现
- **功能**: 检查AI服务是否可用

**请求示例**:
```typescript
const response = await fetch(`${API_BASE_URL}/ai/health`);
const result = await response.json();
```

**响应格式**:
```json
{
  "code": 200,
  "message": "服务正常",
  "data": {
    "status": "healthy",
    "models": ["gpt-3.5-turbo", "gpt-4"],
    "timestamp": "2025-01-XX..."
  }
}
```

### 2. 文档管理接口（适配器）

#### 2.1 创建文档
- **前端调用**: `POST /api/documents/`
- **后端实现**: ✅ 已实现（适配器映射到 `/api/v1/works/`）
- **功能**: 创建新文档（实际创建为作品）

**请求示例**:
```typescript
const response = await apiClient.createDocument(
  userId,
  title,
  content,
  memCubeId
);
```

**后端映射**: 自动转换为作品（Work）创建

#### 2.2 获取文档列表
- **前端调用**: `GET /api/documents/?user_id=xxx`
- **后端实现**: ✅ 已实现（适配器映射到 `/api/v1/works/`）
- **功能**: 获取用户的文档列表

**请求示例**:
```typescript
const docs = await apiClient.listDocuments(userId);
```

#### 2.3 获取单个文档
- **前端调用**: `GET /api/documents/{doc_id}?user_id=xxx`
- **后端实现**: ✅ 已实现（适配器映射到 `/api/v1/works/{id}`）
- **功能**: 获取单个文档详情

#### 2.4 更新文档
- **前端调用**: `PUT /api/documents/{doc_id}?user_id=xxx`
- **后端实现**: ✅ 已实现（适配器映射到 `/api/v1/works/{id}`）
- **功能**: 更新文档信息或内容

#### 2.5 删除文档
- **前端调用**: `DELETE /api/documents/{doc_id}?user_id=xxx`
- **后端实现**: ✅ 已实现（适配器映射到 `/api/v1/works/{id}`）
- **功能**: 删除文档

### 3. 作品管理接口（原生）

#### 3.1 创建作品
- **接口**: `POST /api/v1/works/`
- **功能**: 创建新作品
- **认证**: 需要JWT token

#### 3.2 获取作品列表
- **接口**: `GET /api/v1/works/`
- **功能**: 获取作品列表（支持分页、筛选、排序）
- **认证**: 需要JWT token

#### 3.3 获取单个作品
- **接口**: `GET /api/v1/works/{id}`
- **功能**: 获取作品详情
- **认证**: 需要JWT token

### 4. 章节管理接口

#### 4.1 创建章节
- **接口**: `POST /api/v1/chapters/`
- **功能**: 创建新章节
- **认证**: 需要JWT token

#### 4.2 获取章节列表
- **接口**: `GET /api/v1/chapters/?work_id=xxx`
- **功能**: 获取作品的章节列表
- **认证**: 需要JWT token

### 5. 认证接口

#### 5.1 用户注册
- **接口**: `POST /api/v1/auth/register`
- **功能**: 注册新用户

#### 5.2 用户登录
- **接口**: `POST /api/v1/auth/login`
- **功能**: 用户登录，获取访问令牌

#### 5.3 刷新令牌
- **接口**: `POST /api/v1/auth/refresh`
- **功能**: 刷新访问令牌

## 🔧 CORS配置

后端已配置CORS，允许所有来源访问（开发环境）：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 生产环境应该限制具体域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**生产环境建议**:
```python
allow_origins=[
    "https://yourdomain.com",
    "https://www.yourdomain.com",
]
```

## 🔄 接口映射关系

| 前端接口 | 后端接口 | 说明 |
|---------|---------|------|
| `POST /api/documents/` | `POST /api/v1/works/` | 通过适配器映射 |
| `GET /api/documents/` | `GET /api/v1/works/` | 通过适配器映射 |
| `GET /api/documents/{id}` | `GET /api/v1/works/{id}` | 通过适配器映射 |
| `PUT /api/documents/{id}` | `PUT /api/v1/works/{id}` | 通过适配器映射 |
| `DELETE /api/documents/{id}` | `DELETE /api/v1/works/{id}` | 通过适配器映射 |
| `POST /ai/analyze-chapter` | `POST /ai/analyze-chapter` | 直接对接 |
| `GET /ai/health` | `GET /ai/health` | 直接对接 |

## ⚠️ 注意事项

1. **认证**: 部分接口需要JWT token，需要在请求头中添加：
   ```
   Authorization: Bearer <token>
   ```

2. **用户ID**: 文档接口适配器会将字符串用户ID转换为整数，如果转换失败，默认使用用户ID=1

3. **内容存储**: 文档的内容实际存储在章节（Chapter）中，文档接口会从第一个章节获取内容

4. **流式响应**: AI分析接口使用Server-Sent Events (SSE)格式，前端需要使用EventSource或fetch的流式读取
