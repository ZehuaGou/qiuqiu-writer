# 冲突合并改进说明

## 问题

两个用户同时编辑时，仍然会出现覆盖问题。

## 根本原因

1. **返回内容错误**：后端返回的是客户端原始内容，而不是合并后的内容
2. **合并策略不够智能**：简单的合并可能丢失内容
3. **前端未应用合并结果**：收到合并后的内容但没有更新编辑器

## 已修复的问题

### 1. 后端返回合并后的内容 ✅

**修复前**:
```python
return {
    "success": True,
    "content": content,  # ❌ 返回客户端原始内容
    "version": document["version"],
}
```

**修复后**:
```python
return {
    "success": True,
    "content": document["content"],  # ✅ 返回合并后的内容
    "version": document["version"],
}
```

### 2. 改进合并策略 ✅

**HTML 合并改进**:
- 提取所有块级元素（段落、标题等）
- 使用文本哈希去重，确保不丢失唯一内容
- 保留所有客户端独有的块
- 提取客户端独有的文本节点

**文本合并改进**:
- 按行合并，保留所有唯一行
- 确保合并后的内容包含大部分原始内容
- 验证合并结果，避免异常

### 3. 前端应用合并结果 ✅

**修复前**:
- 收到合并后的内容但没有更新编辑器

**修复后**:
- 检测到内容合并时，自动更新编辑器
- 如果用户正在编辑，延迟应用
- 触发文档更新回调

## 合并策略详解

### HTML 内容合并

1. **提取块级元素**
   - 匹配：`<p>`, `<h1-h6>`, `<li>`, `<div>`, `<blockquote>` 等
   - 提取每个块的文本内容作为唯一标识

2. **去重合并**
   - 使用文本内容的哈希值去重
   - 保留所有唯一的块
   - 如果文本相同但格式不同，保留较长的 HTML（可能包含更多格式）

3. **保留客户端新增**
   - 找出客户端独有的文本节点
   - 提取包含这些文本的块
   - 添加到合并结果中

### 文本内容合并

1. **按行分割**
   - 将内容按行分割
   - 使用行的文本内容作为唯一标识

2. **合并唯一行**
   - 先添加服务器行
   - 再添加客户端独有的行
   - 确保不丢失任何唯一内容

3. **验证结果**
   - 合并后的内容应该包含大部分原始内容
   - 如果异常，使用备用策略（拼接或保留较长版本）

## 测试场景

### 场景1：用户A删除，用户B新增

**步骤**:
1. 初始内容：`<p>Hello</p><p>World</p>`
2. 用户A删除第二个段落：`<p>Hello</p><p></p>`
3. 用户B在末尾添加：`<p>Hello</p><p>World</p><p>New</p>`
4. 用户A先同步 → 服务器：`<p>Hello</p><p></p>`
5. 用户B同步时：
   - 检测到版本冲突
   - 合并：保留 `<p>Hello</p>` + `<p>World</p>` + `<p>New</p>`
   - ✅ 删除操作保留（空段落），新增也保留

### 场景2：两个用户同时新增不同内容

**步骤**:
1. 初始内容：`<p>Start</p>`
2. 用户A添加：`<p>Start</p><p>A的内容</p>`
3. 用户B添加：`<p>Start</p><p>B的内容</p>`
4. 用户A先同步 → 服务器：`<p>Start</p><p>A的内容</p>`
5. 用户B同步时：
   - 检测到版本冲突
   - 合并：`<p>Start</p><p>A的内容</p><p>B的内容</p>`
   - ✅ 两个新增都保留

### 场景3：用户A修改，用户B删除

**步骤**:
1. 初始内容：`<p>Hello World</p>`
2. 用户A修改为：`<p>Hello Universe</p>`
3. 用户B删除为：`<p></p>`
4. 用户A先同步 → 服务器：`<p>Hello Universe</p>`
5. 用户B同步时：
   - 检测到版本冲突
   - 合并：可能保留 `<p>Hello Universe</p>`（因为服务器版本更新）
   - ⚠️ 这种情况需要操作级同步才能完美解决

## 调试方法

### 1. 查看后端日志

```bash
# 查看合并日志
grep "检测到版本冲突\|内容已合并\|HTML 合并完成" backend.log
```

**关键日志**:
```
检测到版本冲突: 服务器版本 X > 客户端版本 Y
服务器内容长度: XXX, 客户端内容长度: YYY
HTML 合并完成：服务器 N 个块，客户端 M 个块，合并后 K 个块
内容已合并，新版本: Z, 合并后长度: LLL
```

### 2. 查看前端控制台

**关键日志**:
```
⚠️ [同步] 检测到服务器版本更新: { serverVersion, clientVersion }
🔄 [同步] 检测到内容合并: { originalLength, mergedLength, version }
✅ [同步] 服务器已合并内容
🔄 [IntelligentSync] 检测到内容合并
✅ [IntelligentSync] 应用合并后的内容
```

### 3. 验证合并结果

检查合并后的内容是否包含：
- ✅ 服务器内容的所有唯一块
- ✅ 客户端内容的所有唯一块
- ✅ 合并后的长度应该 >= max(服务器长度, 客户端长度) * 0.8

## 当前限制

### 1. HTML 结构复杂时

- 如果 HTML 结构非常复杂（嵌套很深），可能无法完美提取所有块
- 解决方案：使用备用策略（拼接或保留较长版本）

### 2. 删除冲突

- 如果用户A删除，用户B修改同一位置，可能无法完美合并
- 解决方案：需要操作级同步（OT）

### 3. 格式信息

- 合并时可能丢失一些格式信息（如样式、属性）
- 解决方案：改进块提取逻辑，保留更多格式信息

## 改进建议

### 短期改进

1. **增强 HTML 块提取**
   - 支持更多 HTML 标签
   - 保留块的所有属性
   - 处理嵌套结构

2. **改进合并验证**
   - 验证合并后的内容是否包含所有原始内容
   - 如果验证失败，使用更保守的策略

3. **添加合并统计**
   - 记录合并成功率
   - 记录合并后的内容长度变化
   - 用于优化合并策略

### 长期改进

1. **实现操作级同步（OT）**
   - 使用已创建的操作跟踪器
   - 实现完整的 OT 转换算法
   - 精确处理所有并发操作

2. **保存历史版本**
   - 实现真正的三路合并（base + server + client）
   - 更准确的冲突检测和解决

3. **用户交互**
   - 当合并结果不理想时，提示用户
   - 让用户选择合并策略
   - 显示合并预览
